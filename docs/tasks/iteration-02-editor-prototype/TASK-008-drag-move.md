# TASK-008 — перетаскивание node мышью на canvas

## Зависимость

TASK-007 завершена. Сначала выполните её проверки, включая визуальную.

## Цель

Выбранный node можно перемещать мышью по canvas; координаты в inspector обновляются синхронно.

## Зафиксированные решения

- Drag начинается по `pointerdown` на node (тот же обработчик, что selection: клик выбирает node и сразу позволяет тащить).
- Пересчёт координат — через PixiJS, а не вручную: новая позиция = `parentView.toLocal(event.global)` минус зафиксированный при `pointerdown` offset (`parentView.toLocal(event.global) - node position` на старте). Это корректно работает при масштабированном/повёрнутом родителе — не пишите собственную тригонометрию.
- Во время drag координаты пишутся в store через существующий `updateNode` на **каждый** `pointermove`. Полный rebuild сцены на каждое движение — ожидаемое поведение (решение итерации); если на sample project движение заметно дёргается, допустимо во время drag двигать display object напрямую, а в store записывать только на `pointerup` — но тогда явно опишите это в отчёте.
- Записываемые `x`/`y` округляются до 2 знаков.
- `pointerup` и `pointerupoutside` завершают drag. Escape отменять не нужно.

## Что создать

1. Drag-логика в `SceneCanvas` (или отдельный модуль `src/canvasDrag.ts`): состояние «идёт drag» локально, не в store.
2. Слушатели `pointermove`/`pointerup` вешаются на `app.stage` (`eventMode: "static"`, `hitArea = app.screen`) только на время drag.
3. Рамка выделения следует за node во время drag.

Не реализуйте snapping, guides, перетаскивание в дереве, resize/rotate handles и drag нескольких nodes.

## Обязательные проверки

```powershell
pnpm build
pnpm typecheck
pnpm test
```

## Визуальная проверка

1. Перетащите серый прямоугольник — он следует за курсором без рывков и «отставания» от мыши.
2. Во время (или после — в зависимости от выбранного режима записи) drag поля `x`/`y` в inspector обновляются.
3. Отпустите кнопку вне node и снова начните drag — поведение корректно, «прилипшего» к курсору node нет.
4. Перетащите text-node внутри container — координаты остаются в системе родителя (проверьте: после drop значение `x` в inspector соответствует видимой позиции).
5. В console нет ошибок и warning от validation.

## Критерии приёмки

- Позиция считается через `toLocal` родителя, без ручных матричных вычислений.
- Итоговые координаты попадают в документ через `updateNode` (валидация сохраняется).
- Пять шагов визуальной проверки воспроизводятся.
- Обязательные команды зелёные.

## Отчёт исполнителя
