# Итерация 04 — реальные image-assets

## Зачем нужна эта итерация

Редактор умеет ориентации и viewports, но image nodes до сих пор рендерятся серыми прямоугольниками: у sample-документа нет реального файла `assets/sample-logo.png`, runtime не загружает текстуры, а Asset Library отсутствует. Эта итерация делает ассеты настоящими: image nodes показывают реальные картинки, появляется панель Assets с upload, у image node можно выбрать другой asset, а замена source и удаление asset получают правильную семантику (ASSET-01, ASSET-02, ASSET-03 и часть RT-03 из плана MVP).

Spine (ASSET-04/05) сознательно отложен на следующую итерацию: сначала image-цепочка должна работать сквозь редактор целиком.

## Главное правило итерации

**Каждая задача заканчивается результатом, который можно открыть в браузере и проверить глазами.** Раздел «Визуальная проверка» обязателен к выполнению и к описанию в отчёте исполнителя.

## Зафиксированные решения итерации

Приняты владельцем проекта; исполнители не выбирают альтернативы и не «улучшают» их:

- **Схема не меняется.** `Asset.source.uri` — уже строка; загруженные в редакторе изображения хранятся как **data URI** (`data:image/png;base64,…`) прямо в документе. Это временное решение прототипа (сервера нет), лимит файла — 2 МБ (localStorage). Фиксируется в ADR 0004.
- `buildSceneView` остаётся **синхронным** и получает опциональную карту готовых текстур `textures: ReadonlyMap<assetId, Texture>`. Загрузка текстур — отдельная async-функция `loadSceneTextures` в runtime-pixi. Отсутствующая или не загрузившаяся текстура **никогда не ломает рендер** — image node рисуется текущим серым плейсхолдером.
- Текстура загружается из URL через `HTMLImageElement` + `decode()` + `Texture.from(image)` — одинаково работает для `data:` URI и для bundled URL. Headless-тесты texture-загрузку не вызывают и передают `Texture.WHITE`.
- Sample logo — реальный PNG в `examples/sample-project/assets/sample-logo.png`; editor-web получает его URL статическим import-ом Vite и резолвит через карту `uri → URL`.
- Кэш текстур в editor-web — модульный `Map<source.uri, Texture>`; старые текстуры при замене source не уничтожаются (осознанное упрощение прототипа).
- «Выбрать другой asset» и «заменить source у asset» — **раздельные операции** (`setImageNodeAsset` и `replaceAssetSource`), как в ASSET-02.
- Удаление используемого asset невозможно: `commitCandidate` с `validateProjectDocument` отбрасывает такой документ по `MISSING_ASSET_REFERENCE`, а UI дополнительно блокирует кнопку по usage count.
- Все изменения документа — только через операции store с `commitCandidate`; невалидный кандидат отбрасывается с `console.warn`.
- Сцена по-прежнему полностью пересоздаётся при любом изменении документа.
- Команды запускаются из PowerShell; `pnpm build` обязан оставаться зелёным после каждой задачи.

## Демонстрируемый результат

`pnpm --filter @pixi-ui-editor/editor-web dev` открывает редактор, в котором:

1. Logo рендерится реальной картинкой (не серым прямоугольником) в обеих ориентациях; сломанный или отсутствующий asset откатывается к плейсхолдеру без ошибок рендера;
2. в левой панели есть раздел Assets с превью и именами всех ассетов документа; кнопка Upload добавляет PNG/JPEG/WebP как data URI, ассет переживает перезагрузку страницы (localStorage);
3. у выбранного image node в Inspector есть секция Image с выпадающим списком ассетов — выбор другого asset немедленно меняет текстуру на canvas;
4. Replace у asset обновляет картинку во всех nodes, использующих его; Delete у используемого asset заблокирован (с подсказкой usage), у неиспользуемого — работает.

## Общий Definition of Done

- `pnpm build`, `pnpm typecheck`, `pnpm test` завершаются с кодом `0` (в этом порядке);
- все четыре пункта демонстрируемого результата воспроизводятся вручную;
- `packages/schema` не изменён; `packages/runtime-pixi` изменён только в объёме TASK-014;
- невалидный документ никогда не попадает в store (подтверждено существующими и новыми тестами store);
- ADR 0004 записан, секция «Текущее состояние» в `agents.md` обновлена.

## Не входит в итерацию

- Spine: загрузка skeleton/atlas, Spine adapter, preview animation (следующая итерация);
- backend, object storage, любые сетевые загрузки ассетов;
- drag-and-drop из панели Assets на canvas, переименование assets, дубликаты, папки;
- освобождение/уничтожение старых текстур, оптимизация памяти;
- миграции схемы, изменение `Asset.source` (versioning-модель остаётся как есть);
- undo/redo, prefabs, exporter.

Если для выполнения задачи приходится реализовывать что-либо из этого списка, остановитесь: scope выбран неверно.
