# TASK-010 — переключение ориентации canvas

## Зависимость

Итерация 02 завершена. Перед началом выполните её обязательные проверки (`pnpm build`, `pnpm typecheck`, `pnpm test`) и убедитесь, что редактор открывается и рендерит sample project.

## Цель

В toolbar появляется кнопка поворота ориентации (как в zStudio). Canvas переключается между горизонтальной раскладкой (профиль `desktop`) и вертикальной (профиль `mobile`): меняются размер артборда и resolved-позиции nodes.

## Зафиксированные решения

- В zustand store добавляется editor-state (не часть документа):

```ts
activeProfile: LayoutProfileId; // начальное значение "desktop"
setActiveProfile(profile: LayoutProfileId): void;
```

`activeProfile` не сериализуется в документ и не сохраняется в localStorage.

- UI-подписи — **Horizontal** (= `desktop`) и **Vertical** (= `mobile`); слова desktop/mobile в UI не показываются.
- Кнопка — одна, toggle в toolbar рядом с существующими действиями; показывает текущую ориентацию (например `⟳ Horizontal` / `⟳ Vertical`) и по клику переключает на противоположную.
- `SceneCanvas` использует `activeProfile` вместо захардкоженного `"desktop"`: артборд рисуется по `scene.layout.referenceViewports[activeProfile]`, сцена строится через `buildSceneView(document, sceneId, activeProfile)`.
- При переключении ориентации камера заново выполняет Fit (артборд другого размера должен целиком попасть в кадр); рамка выделения перерисовывается.
- `updateNode`, drag и inspector в этой задаче **не меняются**: правки по-прежнему пишут базовый `transform`. То, что в вертикальной ориентации правка двигает и горизонтальную раскладку, — известное ограничение, его чинит TASK-011. Не пытайтесь исправить это здесь.

## Что создать

- Поля `activeProfile`/`setActiveProfile` в `apps/editor-web/src/store.ts`.
- Кнопку-toggle в toolbar (`App.tsx`).
- Прокидывание `activeProfile` в `SceneCanvas` (проп или чтение из store) и его использование в двух местах: артборд и `buildSceneView`. Effect пересборки сцены должен зависеть от `activeProfile`.

Новые тесты не нужны: операция не меняет документ.

## Обязательные проверки

```powershell
pnpm build
pnpm typecheck
pnpm test
```

## Визуальная проверка

1. Откройте редактор — canvas в горизонтальной ориентации 1920×1080, как раньше.
2. Нажмите кнопку ориентации → артборд стал вертикальным 390×844 и целиком виден (Fit сработал).
3. Logo в вертикальной ориентации стоит по mobile-override из sample (x=35, y=80, 320×160) — заметно уже и выше, чем в горизонтальной.
4. Выберите node в вертикальной ориентации → рамка выделения совпадает с его вертикальной позицией.
5. Переключитесь обратно → горизонтальная раскладка не изменилась.

## Критерии приёмки

- `activeProfile` отсутствует в сериализованном документе (localStorage-значение `pixi-ui-editor:document` не содержит его).
- Пять шагов визуальной проверки воспроизводятся.
- `packages/schema` и `packages/runtime-pixi` не изменены.
- Обязательные команды зелёные.

## Отчёт исполнителя

Выполнено: `activeProfile` и `setActiveProfile` находятся в zustand editor-state и не входят в сериализуемый `ProjectDocument`. Кнопка `⟳ Horizontal / Vertical` переключает профиль; `SceneCanvas` строит артборд и scene view по активному profile, выполняет Fit при его смене и перерисовывает selection.

Проверки: `pnpm build`, `pnpm typecheck`, `pnpm test` завершились успешно. Ручная визуальная проверка в этом окружении не выполнена: установлен Node.js 20.11.0, а Vite dev требует 20.19+ или 22.12+ и завершается с `crypto.hash is not a function`.
