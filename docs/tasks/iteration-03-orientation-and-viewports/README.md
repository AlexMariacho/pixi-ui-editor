# Итерация 03 — ориентация экрана и размеры viewport

## Зачем нужна эта итерация

Итерация 02 дала работающий редактор, но он рендерит только профиль `desktop` и жёстко фиксирует размеры артборда. Схема и runtime уже умеют два layout-профиля (`desktop`/`mobile`) с overrides — редактор этого просто не показывает. Эта итерация делает mobile-профиль видимым и редактируемым: кнопка поворота ориентации как в zStudio, per-node видимость по ориентациям и настройка размеров экрана в верхней панели.

Соответствие плану MVP: частично ED-«desktop/mobile authoring» (см. следующий продуктовый фронт в `agents.md`). Schema и runtime-pixi **не меняются** — вся работа в `apps/editor-web`.

## Главное правило итерации

**Каждая задача заканчивается результатом, который можно открыть в браузере и проверить глазами.** Раздел «Визуальная проверка» обязателен к выполнению и к описанию в отчёте исполнителя.

## Зафиксированные решения итерации

Приняты владельцем проекта; исполнители не выбирают альтернативы и не «улучшают» их:

- Терминология UI — **Horizontal / Vertical**; в документе это существующие профили: `horizontal ↔ desktop`, `vertical ↔ mobile`. Новых профилей (tablet и т.п.) в схеме не появляется.
- Активная ориентация — **editor-state** (`activeProfile: LayoutProfileId` в zustand store), в `ProjectDocument` не сериализуется и в localStorage не сохраняется; после перезагрузки редактор открывается в horizontal.
- Правки transform идут в активную ориентацию: в horizontal — в базовый `transform`, в vertical — в `layoutOverrides.mobile.transform` (zStudio-паттерн «значения хранятся для active orientation»). `layoutOverrides.desktop` редактор не пишет.
- Per-node видимость по ориентациям — это `layoutOverrides.{desktop|mobile}.visible`; базовый `node.visible` остаётся общим выключателем и не трогается.
- Все изменения документа — только через операции store с `validateProjectDocument` (паттерн `commitCandidate`); невалидный кандидат отбрасывается с `console.warn`.
- Сцена по-прежнему полностью пересоздаётся при любом изменении документа и при переключении ориентации.
- Команды запускаются из PowerShell; `pnpm build` обязан оставаться зелёным после каждой задачи.

## Демонстрируемый результат

`pnpm --filter @pixi-ui-editor/editor-web dev` открывает редактор, в котором:

1. кнопка в toolbar переключает canvas между горизонтальной (1920×1080) и вертикальной (390×844) ориентацией; Logo в вертикальной ориентации стоит по своему mobile-override из sample;
2. перетаскивание и правка transform в вертикальной ориентации меняют только вертикальную раскладку, горизонтальная остаётся нетронутой;
3. в inspector над Transform есть секция Orientation с чекбоксами Horizontal и Vertical; снятая галочка прячет node в соответствующей ориентации;
4. в toolbar есть раздел Screen с пресетами Desktop/Tablet/Mobile и полями W/H, которые меняют размер артборда активной ориентации и сохраняются в документе.

## Общий Definition of Done

- `pnpm build`, `pnpm typecheck`, `pnpm test` завершаются с кодом `0` (в этом порядке);
- все четыре пункта демонстрируемого результата воспроизводятся вручную;
- `packages/schema` и `packages/runtime-pixi` не изменены (кроме случаев, явно указанных в задачах, — таких нет);
- невалидный документ никогда не попадает в store (подтверждено существующими и новыми тестами store);
- ADR 0003 записан, секция «Текущее состояние» в `agents.md` обновлена.

## Не входит в итерацию

- третий layout-профиль (tablet) в схеме и вообще любые изменения схемы;
- responsive-правила, anchors, автоматический пересчёт layout между ориентациями;
- редактирование `settings.layoutProfileSelection` (`mobileMaxAspectRatio`);
- UI сброса override («вернуть как в horizontal») и индикация «поле переопределено»;
- undo/redo, одновременный показ двух ориентаций (side-by-side preview);
- изменения в runtime-pixi, runtime-demo, exporter.

Если для выполнения задачи приходится реализовывать что-либо из этого списка, остановитесь: scope выбран неверно.
