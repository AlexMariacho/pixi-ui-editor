# Стратегия тестирования

Этот документ определяет, какие автоматические тесты нужны Pixi UI Editor на стадии прототипа и как не превратить их в отдельный дорогой продукт. Исторические требования в `docs/tasks/` описывают выполненные итерации; для новых решений применяется эта стратегия и краткая политика из `AGENTS.md`.

## Цель

Тест должен защищать стабильный контракт или риск потери данных. Количество тестов, строк и coverage сами по себе не являются целью. Если регрессию быстрее и надёжнее обнаруживает typecheck, schema validation или короткий browser smoke, отдельный unit-тест не нужен.

## Когда добавлять тест

Добавляйте тест, если выполняется хотя бы одно условие:

- меняется versioned document contract, migration, deterministic serialization или machine-readable validation code;
- операция может оставить document, hierarchy, references, assets, prefabs или bindings в неконсистентном состоянии;
- меняется shared resolver или runtime boundary, от которого должны одинаково зависеть editor preview и игра;
- меняется граница load/save/export/publish, где ошибка может повредить или сделать невоспроизводимым package;
- исправляется регрессия со стабильным входом и наблюдаемым результатом, которую легко повторить без привязки к UI-вёрстке;
- появляется новая существенная ветка логики, не покрытая существующим позитивным или негативным кейсом.

Перед добавлением теста ответьте:

1. Какая реальная поломка будет обнаружена?
2. Стабильный ли это публичный результат, а не деталь реализации?
3. Находится ли тест в самом нижнем слое, владеющем контрактом?
4. Не проверяет ли это уже schema, typecheck или существующий integration test?
5. Можно ли покрыть риск одним репрезентативным кейсом вместо матрицы?

Если на первые два вопроса нет конкретного ответа, тест не добавляйте.

## Что тестировать по слоям

### `packages/schema`

- structure и semantic validation;
- стабильные validation codes;
- references, hierarchy cycles и duplicate IDs/bindings;
- deterministic serialization;
- одну fixture-миграцию для каждого реально поддерживаемого перехода версии.

Полной матрицей ошибок владеет schema. Верхние слои проверяют только то, что ошибка корректно проходит через их boundary.

### `packages/runtime-pixi`

- load/parse boundary и различимость malformed JSON от validation error;
- общий profile resolver, включая обе стороны и точку breakpoint;
- mapping документа в production runtime там, где результат можно проверить без хрупкого pixel snapshot;
- один representative validation error, доказывающий интеграцию со schema.

Не дублируйте здесь все негативные schema cases.

### `apps/editor-web`

- document-команды store, которые изменяют hierarchy, profiles, assets, prefabs или scenes;
- инвариант «валидный document после успешной команды»;
- отказ команды без частичной мутации document;
- load/save preferences только на устойчивых boundary-кейсах вроде повреждённого JSON.

Не тестируйте точную JSX-структуру, CSS, координаты панелей, полный объект UI defaults или тривиальное присваивание transient UI-state. Видимое поведение проверяйте коротким browser smoke по сценарию задачи.

### Export и сквозной pipeline

Поддерживайте небольшое число высокоценных тестов:

- чистый contract `document → package entries`;
- один smoke `document → ZIP → project.json → runtime parse`;
- golden parity editor/runtime добавляйте только когда появится детерминированный способ сравнивать render result без нестабильных screenshot snapshots.

## Ограничение стоимости

- Не вводите coverage quota. Непокрытая строка не является причиной писать тест без продуктового риска.
- По умолчанию достаточно одного позитивного и одного значимого негативного кейса на новую ветку.
- Объём тестового кода в пакете не должен превышать объём продуктового кода; приближение к этому пределу требует пересмотра дублей и слишком мелких тестов.
- Один test-файл должен описывать один домен. Если файл разрастается примерно свыше 250–300 строк или заставляет читать несколько несвязанных подсистем, разделите его и вынесите общую fixture/reset-логику.
- Не создавайте отдельный файл ради одного тривиального теста: учитывайте startup/collection cost Vitest.
- При переносе контракта в нижний слой удаляйте ставшие дублирующими тесты верхнего слоя.

## Текущая организация suites

- schema contract: `packages/schema/src/index.test.ts`, fixture отдельно в `index.test-fixtures.ts`;
- runtime boundary и shared resolver: `packages/runtime-pixi/src/index.test.ts`;
- editor store: `store.assets.test.ts`, `store.hierarchy.test.ts`, `store.prefabs.test.ts`, `store.scenes-and-profiles.test.ts`, `store.selection.test.ts`;
- общий reset/sample для store suites: `store.test-utils.ts`;
- export contract и ZIP smoke: `apps/editor-web/src/exportPackage.test.ts`.

Ссылки на единый `store.test.ts` в завершённых `docs/tasks/` являются историческими отчётами. Не собирайте suites обратно в монолит ради совпадения со старым путём.

## Работа агентов с тестами

- Сначала найдите относящийся к изменению suite через `rg`; не загружайте все тесты репозитория в контекст.
- Читайте общий fixture только если тест действительно от него зависит.
- Для узкого изменения запускайте пакетный или file-scoped тест, затем обязательный корневой `pnpm test`.
- Не обновляйте assertion только ради зелёного результата: сначала подтвердите, что изменился ожидаемый контракт.
- В отчёте указывайте добавленный продуктовый риск и каким минимальным кейсом он покрыт.

## Регулярная ревизия

После крупной итерации быстро просмотрите suites и удалите:

- тесты заглушек и констант;
- дубли нижележащего слоя;
- проверки удалённого поведения;
- assertions на внутреннюю структуру, которые не выражают продуктовый контракт;
- parameter matrix без отдельных веток логики.

Такая ревизия является cleanup, а не требованием увеличивать покрытие.
