# ADR 0012 — Project folder as the workspace exchange format

## Status

Accepted.

## Context

Before iteration 12 the editor lived entirely inside one browser: the document (with asset bytes inlined as data URIs) sat in `localStorage`, capped at roughly 5 MB total and 2 MB per asset, and the only way to hand work to someone else was the immutable runtime export package — not a source a second person could reopen, edit and re-export. Iteration 12 (TASK-045/046/047) introduces a **project** as the unit of authorship: a folder on disk that is a complete, lossless, git-friendly source, plus a per-browser working copy that autosaves every edit without needing a folder at all.

## Decision

- **A project folder is the only exchange format.** Layout: `project.json` (the canonical `ProjectDocument`, existing schema versions and migrations unchanged), `manifest.json`, `editor.json`, `assets/<assetId>/<fileName>`. The `assets/` layout intentionally matches the runtime export package (ADR 0006); asset file naming is one shared module (`packages/schema/src/assetPaths.ts`), not a parallel copy. A single-file bundle (`.pixiproj`/zip) and cloud/server storage are explicitly rejected for this iteration: the folder already covers the git push/pull exchange scenario the project needs, and zip stays reserved for the immutable runtime export.
- **`project.json` is the only required file.** A missing or unparsable `manifest.json` is regenerated with a `console.warn`, never blocks Open; `editor.json` is parsed tolerantly (`parseEditorJson`) and its absence falls back to safe defaults. Save (`writeProjectFolder`) only ever touches `manifest.json`/`project.json`/`editor.json`/`assets/**`; it never deletes or rewrites unrelated files (`.git`, `README.md`, ...), and repeating a Save with no changes produces byte-identical files.
- **The manifest is minimal and versioned separately from the document schema:** `formatVersion`, a stable `projectId`, `name`, `createdAt`, `editorVersion` (the editor build that performed the last save). There is deliberately no "last saved at" field — it would produce a git diff on every Save for no informational gain git itself doesn't already track. Future cloud-facing fields (author, remote id, ...) are new `formatVersion`s, not speculative reserved fields today.
- **`editor.json` holds project-level view context, not personal preferences:** the active scene and the active layout profile (desktop/mobile), so a recipient opens the folder and sees the scene/orientation "as the author left it". Key bindings, floating-window positions/sizes and the Assets window's view mode stay per-user in browser `localStorage` (`shared/uiPrefs.ts`) and are never written into the folder.
- **The document never stores data URIs, on disk or in the working copy.** Every asset URI is a package-relative `assets/<assetId>/<fileName>` path; binary bytes live as `Blob`s in IndexedDB and are resolved to a display `blob:` URL through the existing `AssetUrlResolver`/`assetUrlRegistry`. This removes the old 2 MB per-file cap.
- **The IndexedDB working copy is independent of any folder.** Every committed document/editor-state mutation autosaves into IndexedDB (`shared/projectStore`), so closing the tab loses nothing; the file on disk only changes on an explicit Save. `dirty` means "the working copy differs from the last successful Save/Open", not "unsaved to the browser at all" — it stays `true` for a brand-new project until its first Save As.
- **Exactly one open project at a time**, matching a single-user desktop-app model. New/Open fully replace the working copy; there is no project list, no recents, only **Continue** for whatever working copy is already present. `useEditorStore`'s `projectOpen` flag is intentionally session-scoped, not derived from "does a working copy exist": the editor always starts at a startup screen (New/Open/Continue) on every load, regardless of what IndexedDB already holds, so a user always makes an explicit choice instead of being dropped back into a canvas they didn't ask to see. The bundled `examples/sample-project` is never auto-materialized into a working copy any more — it stays a schema fixture/reference document only.
- **Disk access is lazy File System Access API (Chromium) permission.** The directory handle is remembered in IndexedDB (`putFolderHandle`) so it survives a reload, but `queryPermission`/`requestPermission` are only ever called from an actual Save/Open user gesture (`ensureReadWritePermission`), never on page load. Continue works with zero disk permission because it only touches the working copy. Browsers without the API get an explicit disabled state and message on Open/Save/Save As; New and Continue still work everywhere, since neither touches disk.
- **New Project and the dirty guard share one replace path.** `workspace.slice.ts`'s `applyWorkspaceSnapshot` is the single function both New and Open funnel through: it clears and repopulates the IndexedDB working copy, re-hydrates the asset URL registry from scratch, rebinds (or unbinds) the folder handle, and resets every piece of transient editor state (selection, preset-editing mode, undo/redo history, and per-node authoring records such as spine scrubbing or particle playback/diagnostics) that would otherwise reference node IDs from the previous document. New/Open are reachable both from the startup screen and from a running editor's toolbar (registered as `project.new`/`project.open` in the command registry, ADR 0007); whenever the working copy is dirty, both first show an explicit Save (Save As if unbound) / Discard / Cancel choice — no path silently drops unsaved work.

## Consequences

A designer can Save a project into a folder, commit it to git, and a programmer can clone the repository, Open the same folder, see the same windows/assets/active scene/orientation, and produce an Export package byte-identical to the designer's own. Repeated saves without changes produce no git diff. The trade-off is the scope iteration 12 explicitly defers: no cloud/server storage or sharing-by-link (only a manifest shape that could support it later), no project list/recents/templates, no background disk sync or file-watching for externally-changed folders, no git conflict/merge tooling inside the editor, and the folder workflow is unavailable (with a clear message) on browsers without the File System Access API — only the IndexedDB working copy works everywhere.
