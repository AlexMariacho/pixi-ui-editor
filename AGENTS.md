# Pixi UI Editor — инструкции для агентов

## Контекст проекта

Pixi UI Editor — web-first редактор для сборки игрового UI на PixiJS. Редактор владеет только presentation-структурой, а игровое приложение — поведением и gameplay-логикой.

Перед началом архитектурной или продуктовой работы прочитайте документы в [`docs/`](docs/):

- `game-ui-authoring-platform-concept.md` — целевая продуктовая и архитектурная концепция;
- `game-ui-platform-implementation-plan-mvp-v3.md` — границы MVP, порядок реализации и backlog;
- `zstudio-review-for-game-ui-platform.md` — полезные паттерны и анти-паттерны из zStudio.

## Текущее состояние проекта

Поддерживайте эту секцию актуальной: обновляйте её при завершении итераций и принятии решений, меняющих состояние репозитория.

- **Итерация 01 (Этап 0, технический foundation) завершена**: monorepo, schema v0 (`packages/schema`), headless loader (`packages/runtime-pixi`), эталонный документ `examples/sample-project/project.json`, ADR 0001.
- **Итерация 02 завершена**: `apps/editor-web` — работающий визуальный прототип на Vite/React/zustand/PixiJS. Он рендерит sample через shared runtime, поддерживает selection, inspector, drag, добавление/удаление nodes и восстановление документа из `localStorage`; архитектура зафиксирована в ADR 0002.
- **Итерация 03 завершена**: editor-web переключает Horizontal/Vertical preview через transient `activeProfile`, transform и visibility редактируются независимо для обоих layout-профилей, а раздел Screen в toolbar сохраняет editable reference viewport активной ориентации с Desktop/Tablet/Mobile presets; решения зафиксированы в ADR 0003.
- **Итерация 04 завершена**: editor-web отображает реальные image-textures, позволяет загружать image assets в localStorage, назначать asset image-node, заменять source с сохранением stable ID и удалять только неиспользуемые assets; решения зафиксированы в ADR 0004.
- **Итерация 05 завершена**: editor-web получил плавающее окно Assets с list/grid-представлениями и отдельными UI-preferences, а также viewport-инструменты Pan / Select / Resize с profile-aware resize-ручками; решения зафиксированы в ADR 0005.
- **Итерация 06 завершена**: editor-web поддерживает Spine-ассеты (drag-and-drop, loop и frame scrubbing), несколько окон (Windows) с режимом Map, пресеты (prefab definitions + prefab-instance nodes с режимом редактирования) и кнопку Export, собирающую самодостаточный zip-пакет (`project.json` + `assets/<assetId>/<fileName>`, zip через `fflate`); `packages/runtime-pixi` даёт `resolveProfileForViewport` и `loadSceneView`; демо-приложение `examples/pixi-app` (ванильный Vite + PixiJS, без React) воспроизводит распакованный пакет из `public/package/`; решения зафиксированы в ADR 0006.
- Глобальные действия editor-web проходят через единый command registry; UI и клавиатура вызывают одни команды, а replaceable key bindings хранятся отдельно от реализаций и `ProjectDocument`; решение зафиксировано в ADR 0007.
- **Итерация 07 завершена**: schema/runtime/editor поддерживают profile-aware parent-relative anchors; Inspector получил Unity-подобные 3×3 presets с Shift для синхронизации pivot и Shift+Ctrl для немедленного snap; решение зафиксировано в ADR 0008.
- **Stretch-якоря (schema v2)**: точечный `anchorX/anchorY` заменён Unity-парой `anchorMin*/anchorMax*` с миграцией v1→v2; растянутая ось хранит width/height как дельту к якорному прямоугольнику (Inspector редактирует её как Left/Right/Top/Bottom); пресеты — сетка 4×4 со stretch-рядом/столбцом; выделенная нода всегда показывает гизмосы якорей; все типы нод рендерятся через один базовый класс `NodeView` в runtime-pixi; решение зафиксировано в ADR 0009.
- Toolbar editor-web содержит кнопку Preview: она открывает выбранное окно проекта без editor overlays в отдельном browser popup через `packages/runtime-pixi`, запрашивает половину текущего reference viewport и при ручном resize только масштабирует фиксированный layout-profile. Явная смена ориентации или окна в editor полностью пересоздаёт Pixi scene root открытого Preview с повторным использованием загруженных texture/Spine data.
- Остальные `apps/*` и пакеты, кроме `packages/schema` и `packages/runtime-pixi`, остаются запланированными заглушками; workspace также включает `examples/*`.
- **CI намеренно отсутствует** (решение владельца проекта от июля 2026): GitHub Actions вернём после того, как появится работающее приложение. Не добавляйте workflow без явного запроса.
- React подключён только в `apps/editor-web`; PixiJS — в `apps/editor-web` и `examples/pixi-app`; серверный framework ещё не выбран.

## Рабочий процесс разработки

Требования: Node.js **20.19+ или 22.12+** (рекомендован 22 LTS), pnpm 10 (`npm install -g pnpm@10.27.0`, если нет в PATH). Основная среда разработки — Windows; команды должны работать в PowerShell. Перед запуском editor-web проверьте `node --version`: Vite 7 не запускается на Node.js 20.11.0 и падает с `TypeError: crypto.hash is not a function`, хотя `pnpm build`, `pnpm typecheck` и `pnpm test` могут завершиться успешно.

```powershell
pnpm install --frozen-lockfile
pnpm build       # обязательно ДО typecheck
pnpm typecheck
pnpm test
```

Важно: `pnpm typecheck` на чистом checkout падает до первого `pnpm build`, потому что типы межпакетных импортов резолвятся из `dist/` (`exports.types → ./dist/index.d.ts`). Это известное ограничение; если будете его устранять (project references и т.п.) — обновите README и эту секцию.

Smoke-тест эталонного документа: `pnpm --filter @pixi-ui-editor/runtime-pixi test`. Путь к fixture захардкожен в тесте — не перемещайте `examples/sample-project/` без правки тестов.

Запуск редактора из корня: `pnpm --filter @pixi-ui-editor/editor-web dev`; откройте URL из вывода Vite (обычно `http://localhost:5173/`). Если порт занят, Vite выберет следующий свободный порт. Для проверки production bundle сначала выполните `pnpm build`, затем `pnpm --filter @pixi-ui-editor/editor-web preview`.

## Проектирование новых итераций

Новые итерации оформляйте по образцу [`docs/tasks/iteration-01-foundation/`](docs/tasks/iteration-01-foundation/):

- отдельные файлы `TASK-NNN-*.md` со сквозной нумерацией, каждый с разделами: зависимости, цель, зафиксированные решения, что создать, обязательные проверки, критерии приёмки, отчёт исполнителя;
- зависимости задач оформляйте как явный граф: независимые задачи можно выполнять параллельно, а последовательность обязательна только для названных технических зависимостей;
- feature-задача проверяет собственный контракт targeted-тестами и browser smoke; полный `build` → `typecheck` → `test` выполняется один раз на foundation/integration gate, а не повторяется как preflight каждой следующей задачи;
- новые архитектурно значимые решения фиксируйте отдельными ADR в `docs/adr/` (следующий номер — 0002); ADR описывает только уже принятые решения;
- добавляйте ссылку на новую итерацию в `docs/README.md`;
- scope сверяйте с вопросом из плана MVP: нужно ли это, чтобы дизайнер собрал UI, опубликовал package, а PixiJS runtime воспроизвёл его без ручной пересборки?

## Базовые архитектурные правила

- **Каждый тип ноды, который можно вытащить на сцену, наследуется от `NodeView` (`packages/runtime-pixi`) и получает всё общее поведение только от него.** Базовому классу принадлежат layout-профили, anchors, позиция, pivot, scale, visibility и зона захвата, равная layout-прямоугольнику ноды. Подкласс реализует единственный метод `syncContent(...)` и отвечает исключительно за собственный контент.
- **Визуальные изменения применяйте инкрементально.** Смена texture, текста, tint, visibility, размеров, transform или visual state должна мутировать существующие Pixi views через `NodeView.syncContent(...)` или специализированный runtime API; не пересобирайте и не `destroy`-те сцену, `NodeView` или поддерево ради такого обновления. Полная пересборка допустима только при реальном изменении topology (создание/удаление/перепривязка nodes, смена scene/window/profile) либо когда базовая библиотека не предоставляет безопасного in-place API; исключение и его стоимость должны быть явно зафиксированы. При проектировании visual logic выбирайте минимальный объём затронутых объектов и сохраняйте стабильную identity views, чтобы authoring canvas и Preview не создавали лишнюю работу GPU/GC.
- **Новые asset-ссылки ноды подключайте к общему asset contract.** Добавьте их в исчерпывающий `collectNodeAssetIds(...)`, а view синхронизируйте через общий mutable texture map `NodeView.textureFor(...)`. Тогда editor догрузит ранее неиспользованный asset и применит его к уже существующему view без отдельного loader-а, rebuild или `destroy`.
- **Selection и drag не должны зависеть ни от того, чем нода отрисована, ни от её вложенности.** Пустой `Container` в PixiJS не имеет `containsPoint`, поэтому без зоны захвата базового класса нода находится под курсором только через свой контент: ноды, которые ничего не рисуют (container) или прячут контент от hit-теста (button — его `@pixi/ui` view инертен в authoring), молча перестают выделяться и двигаться. Это не «особенность нового элемента», а регрессия: новый тип ноды обязан быть selectable, draggable и одинаково показывать pivot/anchors сразу, без собственного кода интерактивности.
- **Зона захвата задаётся через `containsPoint(...)` и никогда через `hitArea`.** `hitArea` в PixiJS — не зона захвата, а фильтр поддерева: `hitPruneFn` отбрасывает контейнер вместе со всеми детьми, как только точка выходит за его прямоугольник. Так как `NodeView` держит и собственный контент, и детей, `hitArea` обрезала бы любого ребёнка, вылезающего за границы родителя (а дочерняя нода по умолчанию создаётся со смещением и вылезает). Рендеринг при этом не обрезается, поэтому объект остаётся видимым, но неинтерактивным. `containsPoint` получает точку в локальных координатах, детей не обрезает, и они проверяются раньше и независимо от родителя.
- Отключая интерактивность библиотечного контрола на authoring-канвасе (`eventMode = "none"`), никогда не переносите hit-тест в подкласс: попадание по ноде остаётся ответственностью `NodeView`.
- Инвариант закреплён в `packages/runtime-pixi/src/index.test.ts` двумя тестами: `gives every node type the same grab rectangle, whatever it renders` построен на `Record<UINode["type"], ...>`, поэтому новый тип ноды не пройдёт компиляцию, пока не будет в него добавлен; `keeps a child grabbable where it reaches past its parent's rectangle` держит запрет на `hitArea`. Не ослабляйте эти проверки — они существуют, чтобы вопрос «почему новый элемент не двигается» больше не возникал.
- Документ UI декларативен: не сериализируйте внутренние объекты PixiJS.
- Системная идентичность сущностей строится на неизменяемых stable ID; имена, пути и индексы не являются идентичностью.
- Node и Asset — отдельные сущности. Замена файла ассета не должна неявно ломать ссылки.
- В сцене одна общая hierarchy. `desktop` и `mobile` реализуются как layout-профили с overrides, а не как копии сцен.
- Editor preview и игра должны использовать один production runtime и одинаковый profile resolver.
- Runtime API остаётся generic и не содержит жанровую или игровую семантику. Игровой код подключается через bindings/slots.
- Публикация создаёт версионированный неизменяемый package; draft не является контрактом для игры.

## MVP-приоритеты

Сначала развивайте сквозной сценарий: создание сцены → ассеты (включая Spine) → desktop/mobile layout → prefab → bindings → publish/export → загрузка в PixiJS runtime. Не расширяйте MVP timeline-редактором, behavior graph, готовой библиотекой игровых controls или дополнительными runtime targets без отдельного решения.

## Изменения

- Сохраняйте schema versioning и миграции с первой версии формата.
- Добавляйте validation для ссылок, prefabs, bindings и обоих layout-профилей.
- При изменениях документа, layout resolver или runtime добавляйте тесты, покрывающие desktop и mobile, включая viewport около breakpoint.
- Обновляйте документы в `docs/`, если меняются зафиксированные архитектурные решения или границы MVP.

## Политика тестирования на раннем этапе

Проект находится на стадии прототипа. Пишите только **необходимые** тесты — объём тестового кода не должен превышать объём продуктового кода.

Полные критерии выбора, владения и ревизии тестов описаны в [`docs/testing-strategy.md`](docs/testing-strategy.md). Coverage quota не вводите: каждый новый тест должен называть конкретный продуктовый риск и проверять стабильный контракт минимальным числом кейсов.

Что тестировать обязательно:

- контракт документа: schema, validation codes, миграции, детерминированную сериализацию;
- границы загрузки/сохранения документа (round-trip, различимость ошибок);
- негативные кейсы, у которых есть стабильный machine-readable код ошибки.

Чего избегать:

- тестов на заглушки, тривиальные getters/setters и прямой проброс вызовов;
- дублирующих тестов, проверяющих одно и то же поведение через разные входы без новой ветки логики;
- исчерпывающих матриц параметров там, где достаточно одного репрезентативного позитивного и одного негативного кейса;
- тестов UI-вёрстки и внутренних деталей реализации, которые ломаются при любом рефакторинге.

Дополнительные правила:

- негативной матрицей владеет самый нижний слой контракта; верхние слои проверяют один representative error на своей границе, а не копируют все schema cases;
- по умолчанию достаточно одного позитивного и одного значимого негативного кейса на новую ветку логики;
- не сравнивайте целиком растущие объекты UI defaults, если контрактом является безопасный fallback;
- разделяйте test-файл по доменам, когда он приближается к 250–300 строкам или заставляет агента читать несвязанные подсистемы; общие fixtures/reset helpers выносите отдельно;
- не дробите suite до одного тривиального теста на файл: учитывайте startup/collection cost Vitest;
- после крупной итерации удаляйте тесты заглушек, дубли нижнего слоя и assertions, которые больше не выражают продуктовый контракт;
- для видимого UI используйте browser smoke по сценарию задачи, а не DOM/CSS-тесты.

Если задача в `docs/tasks/` явно требует конкретный список тестов — этот список и есть «необходимый» объём: реализуйте его, но не расширяйте по собственной инициативе.
